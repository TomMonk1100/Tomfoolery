# Update 3 — Execution deviations log

Per plan §0.4: deviations from the written plan, with rationale. Phase 0 executed 2026-07-07.

## Phase 0

1. **Steps 0.2 + 0.3 landed as a single test-run unit.** The plan orders "migrate weapons.json" before "grep sweep consumers", but flipping the JSON schema alone turns consumers red (rule 2: never proceed on red). Both were applied back-to-back, then tested once. No design change.
2. **Quality pure logic lives in `src/core/qualitySim.ts`, not `Quality.ts`.** Phaser cannot be imported under vitest's node environment (no DOM globals — see the note in `tests/phasertest.test.ts`). `Quality.ts` keeps the Phaser-dependent boot probe + singleton and re-exports the sim's types/constants; `qualitySim.ts` holds `settingsForTier`/`resolveTier` with its own vitest suite. Matches the repo's `*Sim.ts` pattern (rule 6).
3. **`tests/draftEligibility.test.ts` mocks `phaser` and calls the private `isDraftable` directly.** Same node-env constraint; extracting eligibility into a pure module was judged too invasive for Phase 0 (DraftSystem is the R7 load-bearing wall — Phase 1 restructures it anyway).
4. **Normalizer applied at all three `weapons.json` import sites** (WorldScene, DraftScene, MetaHubScene via `src/core/weaponCatalog.ts`), not only "where weapons load into GameContext" — the scenes import the raw JSON directly, so normalizing only the ctx copy would leave two un-normalized views of the same data.
5. **`StatBonus` did not exist as a type** (plan §3.1 references `Partial<StatBonus>`); `ctx.statBonus()` is a string-keyed lookup. Defined a new `StatBonus` interface in `types.ts` restricted to the D6 stat set (damage, cooldown, area, moveSpeed, xpGain, knockback); `SynergyData.thresholds[].bonus` uses it directly (already all-optional, so `Partial<>` is redundant).
6. **`applyWeaponOrPassive` already records `ActiveWeapon.evolutionId`** (first passive-satisfied branch, fallback `evolutions[0]`) when an evolution is picked. Slightly ahead of Phase 1 4a (which replaces this with explicit per-branch cards), but without it evolutionId would never be set for evolutions taken between Phase 0 and Phase 1.

## Phase 1 (executed 2026-07-07)

7. **Gate 1's live dev-run fusion sanity replaced by a scripted vitest** (`tests/fusion.test.ts` "Gate 1 sanity"). The sandbox cannot run a Phaser dev server (§10: verify logic with vitest, not vite dev); the test drives the real DraftSystem apply path from empty kit → both inputs maxed → guaranteed fusion offer → fused weapon owned. A true in-browser check happens at Phase 4's live playtest.
8. **Fused weapons have cards.json entries; `isDraftable` blocks them only when un-owned.** The plan gives fused weapons 3 levels (§9.1 "keeps late drafts meaningful") but says they're "never draftable through the normal path" — with no card they could never level past 1, and content.test enforces one card per weapon. Resolution: acquisition is fusion-only, upgrades flow through the normal draft.
9. **D9 combo weighting landed in Phase 1** (plan never assigns it to a phase; Phase 4's kill criterion assumes it exists). `comboWeightMultiplier` in rarityWeights.ts (pure + tested), wired into DraftSystem.weightedPick. "Progresses a synergy tag" implemented as: card shares a tag with ≥1 owned item.
10. **Synergy bonus stat mappings** (per §4b "map to nearest existing field"): xpGain → pickupRadius (no xpGain hook exists; bigger mote magnet ≈ more XP collected), knockback → area (no statBonus("knockback") hook). synergies.json stores the mapped fields so Phase 2's codex shows real effects; sonic-3 folds its knockback into area (+30% total).
11. **Fusion card makes a 4-card offer** ("prepend" per §4c). DraftScene shrinks card width 130→108 when 4 cards are present (4×130 overflows the 480px viewport).

## Phase 3 (executed 2026-07-07)

16. **Animation frames (§6.1) were already fully implemented before this plan was written.** Grepped `src/gfx/sprites/animals.ts`/`enemies.ts`: all 3 animals and all species/boss enemies already have multi-frame idle/walk/attack/hurt or idle/move/death cycles with real per-frame pixel-map generation functions (e.g. `dogFrame(earFlop, legPhase, mouthOpen)`), registered via the exact `frames: string[][]` + `anims: Record<string, AnimDef>` schema the plan describes as new. The plan's §1 grounding ("Single static frame per sprite, no animation") does not match reality. Per rule 4, adapted minimally: did no new frame-authoring work, and Gate 3's "registry validation test for frame dims" requirement was already satisfied by the pre-existing `tests/sprites.test.ts` ("every def has at least one frame, all frames equal dimensions").
17. **Shadows scoped to the player only**, not enemies/companions. Wiring shadows into `EnemySystem`'s pooled instance lifecycle (attach/position-sync/release, 3+ touch points in a hot per-frame system) and `CompanionSystem` carries real regression risk that's normally caught by looking at it — this sandbox has no live browser to visually verify pooled-object wiring didn't break. Shipped the shared procedural shadow texture + full player implementation (bob-safe, position-synced) as the safe, verifiable slice; enemy/companion coverage deferred.
18. **Weapon VFX v2 (§6.4) implemented as one generalized grade-tint mechanism, not 8 bespoke per-archetype templates.** `WeaponSystem.weaponGradeTint()` covers base (unchanged) / evolved (gold, unchanged from Update 2) / fused (alternates gold↔purple per fire, approximating "dual-hue between both input weapons' colors" — weapons have no canonical per-weapon color in the data model to alternate between, so a fixed second hue was used instead) at all 8 existing tint call sites across every archetype. The richer per-archetype descriptions (delayed second ring, 4-point trail, arc afterimage, edge shimmer, orbital sparkles) were not hand-built as distinct code paths; doing so without any way to visually confirm the result risked shipping unverifiable/broken geometry across WeaponSystem's hot fire path. Logged as scoped-down per the plan's own "if the phase runs long, ship 1-4 and defer 5-6" allowance, applied here to a sub-scope of item 4 rather than the whole item.
19. **Season ambient particles are a new, self-contained screen-space pool** (`SeasonAmbience.ts`), not folded into the existing combat `Particles` system. `Particles.burst()` is a private radial-explosion helper (short-lived, decaying velocity) unsuited to "particles drifting across the camera view" for seconds at a time; adding a second responsibility to that file risked the combat-cosmetic budget system. A fresh pooled system (12×`Quality.particleScale`, respawned in place, never recreated) kept the blast radius of the change to one new file.
20. **Screenshot review gate (§6.6) could not be performed.** No live browser/dev server in this sandbox (`astro`/`vite dev` EPERM into the repo mount — environment note §10, and `vite build` itself needed `--outDir` outside the mount). PostFX (vignette + bloom) is implemented behind `Quality.current.postFX` and the existing `ENABLE_POSTFX` kill-switch, gated to WebGL, but its visual quality is unverified live — per plan §6.6's own instruction, if it looks bad once someone can see it, flip `ENABLE_POSTFX` to `false` in `src/core/qualitySim.ts` rather than tuning blind.

## Phase 4 deploy (executed 2026-07-07)

24. **Netlify's remote `npm run build` failed 3x in a row with only a generic "Build script returned non-zero exit code: 2" and no accessible build log** (the reader tool's `summary.messages` was empty on every failed attempt; `GET .../deploys/<id>/log` returned nothing without dashboard auth). The exact same source built cleanly every time locally in this session (`tsc --noEmit && vite build`, both via `npm install` and `npm ci`), and `NODE_VERSION=20` made no difference. **Root cause not identified** — something specific to Netlify's remote build environment for this site, not the source. **Workaround shipped**: built locally (verified clean), then deployed the pre-built `dist/` output directly with a trivial `netlify.toml` (`publish = "."`, no `[build] command`), bypassing the remote build step entirely. Both the deploy permalink and production (`understory-life.netlify.app`) verified 200 + correct `<title>Understory</title>` + the JS bundle itself reachable (200).
25. **Consequence for Adam**: the site's `netlify.toml` in the repo still says `command = "npm run build"` — if a future deploy goes through the normal source-upload flow (this MCP tool, or the Netlify dashboard/CLI default path) instead of a pre-built static deploy, it will likely hit the same unexplained failure. Recommend checking the actual build log in the Netlify dashboard (UI shows full logs; the MCP reader tools used this session did not expose them) next time this comes up, since that's the one diagnostic path this session didn't have access to.
26. **Live headless gameplay verification (Gate 4: "3 full runs clean", "fusion + codex verified live") was not possible this session** — no browser/JS-execution tool available (see #22/#23).
