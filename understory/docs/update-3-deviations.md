# Update 3 — Execution deviations log

Per plan §0.4: deviations from the written plan, with rationale. Phase 0 executed 2026-07-07.

## Phase 0

1. **Steps 0.2 + 0.3 landed as a single test-run unit.** The plan orders "migrate weapons.json" before "grep sweep consumers", but flipping the JSON schema alone turns consumers red (rule 2: never proceed on red). Both were applied back-to-back, then tested once. No design change.
2. **Quality pure logic lives in `src/core/qualitySim.ts`, not `Quality.ts`.** Phaser cannot be imported under vitest's node environment (no DOM globals — see the note in `tests/phasertest.test.ts`). `Quality.ts` keeps the Phaser-dependent boot probe + singleton and re-exports the sim's types/constants; `qualitySim.ts` holds `settingsForTier`/`resolveTier` with its own vitest suite. Matches the repo's `*Sim.ts` pattern (rule 6).
3. **`tests/draftEligibility.test.ts` mocks `phaser` and calls the private `isDraftable` directly.** Same node-env constraint; extracting eligibility into a pure module was judged too invasive for Phase 0 (DraftSystem is the R7 load-bearing wall — Phase 1 restructures it anyway).
4. **Normalizer applied at all three `weapons.json` import sites** (WorldScene, DraftScene, MetaHubScene via `src/core/weaponCatalog.ts`), not only "where weapons load into GameContext" — the scenes import the raw JSON directly, so normalizing only the ctx copy would leave two un-normalized views of the same data.
5. **`StatBonus` did not exist as a type** (plan §3.1 references `Partial<StatBonus>`); `ctx.statBonus()` is a string-keyed lookup. Defined a new `StatBonus` interface in `types.ts` restricted to the D6 stat set (damage, cooldown, area, moveSpeed, xpGain, knockback); `SynergyData.thresholds[].bonus` uses it directly (already all-optional, so `Partial<>` is redundant).
6. **`applyWeaponOrPassive` already records `ActiveWeapon.evolutionId`** (first passive-satisfied branch, fallback `evolutions[0]`) when an evolution is picked. Slightly ahead of Phase 1 4a (which replaces this with explicit per-branch cards), but without it evolutionId would never be set for evolutions taken between Phase 0 and Phase 1.

## Post-launch fix #2: camera-wrap regression (2026-07-08, live browser available this session)

32. **Root cause found and fixed: the camera-wrap rewire in #30 introduced two new bugs** ("can walk past the border into black", "grass moves with me oddly"), both traced to the ground TileSprite in `WorldGenSystem.renderTiles()`. It was a `size*tilePx` (4096x4096) quad fixed at world (0,0) and parented as a CHILD of the wrapped statics container (`this.container`) — a design that only worked while the camera was clamped inside the world rect (pre-#30). Once #30 made the camera unbounded to smooth-follow across the seam: (a) the camera's viewport could show area outside that fixed 4096x4096 quad, rendering as black — the "past the border" bug; (b) `WrapRenderSystem`'s per-child statics-wrap loop was ALSO repositioning the ground sprite's x/y every frame (since it was a container child with a numeric x/y), on top of the `setTilePosition` scroll already being applied to it — a double motion causing the "grass moves oddly" drift. Fixed by making the ground sprite genuinely screen-locked: sized to the canvas (`scene.scale.width/height` instead of world size), `scrollFactor(0)`, depth `-1`, and kept OUTSIDE `this.container` as a top-level scene object so the statics wrap loop never touches it (`WrapRenderSystem.ts` also got a defensive `if (child === this.ground) continue` guard in that loop, belt-and-suspenders).
33. **This fix WAS visually verified live**, unlike everything in the Phase 0-4 + fix-round-1 work. The Chrome MCP browser tools were available this session. Verification method: rather than trying to drive the touch-joystick drag gesture through browser automation (unreliable — Phaser's swipe/drag gesture classifier is timing-sensitive and CDP-synthesized drags tend to misclassify), used the game's existing `window.__understory` debug hook (the same one `docs/playtest-checklist.md`'s headless-playtest technique references) to directly set `playerContainer.x/y` to positions straddling the world seam (x=4091, then x=2, out of a 4096px-wide world) and screenshotted each. Both showed a fully-filled, seamless ground with zero black — confirming (a) is fixed. Also directly inspected the live runtime object graph (`scene.world.getRenderRefs().ground`) and confirmed `x:0, y:0, scrollFactorX:0, parentContainer: none` and that its `tilePosition` doesn't drift when idle — confirming (b) is fixed at the code level (the exact broken code path — container-child position wrapping — structurally cannot reach this object anymore).
34. **One caveat**: true continuous-motion panning (camera smoothly catching up frame-by-frame while walking) was NOT captured on video/in real time — the automated browser tab reported `document.visibilityState: "hidden"` (a background tab from the page's own perspective, even though it was the tool's active target), which throttles/freezes Phaser's rAF loop between tool calls. This didn't block verifying the two reported bugs specifically (both are static-position/structural issues, confirmed either instantaneously via the seam screenshots or via direct object-graph inspection), but a genuine live playthrough by Adam is still the strongest confirmation of overall feel.
35. Redeployed same pre-built-`dist` workaround (Netlify's remote build still unexplained-broken, see #24/#25 — unchanged). Deploy `6a4da3e48cc59d03978cb987` reached `state: ready`, `context: production`, `published_at` set; both the permalink and the production alias (`https://understory-life.netlify.app`) curl-verified 200 + correct title. 267 vitest passing, `tsc --noEmit` clean, local `vite build` clean throughout.

## Post-launch fix #3: skunk-cloud / purr-aura zone-stacking bug (2026-07-08)

36. **Adam reported skunk spray rings stacking into a screen-covering mess after ~15 kills, referencing a prior "Update 4" 24-zone cap that supposedly already addressed this.** `git log` on this repo shows no such update — the most recent commits before this session were the two camera-wrap fix rounds (#30-35). No stash, no other branch, no uncommitted diff touching `WeaponSystem.ts` or `weapons.json` existed either. Whatever "Update 4" Adam was recalling did not land in this repo; told him this directly rather than assuming the described cap existed somewhere.
37. **Root cause (confirmed straight from the live deployed bundle, not just source)**: fetched `understory-life.netlify.app`'s shipped JS and grepped the inlined weapon data. `skunk-cloud` (all 5 levels + evolution A "Toxic Fog") and, independently, `purr-aura` (all 5 levels + its evolution) both shipped with `durationMs: 99999` (~100 real seconds) while firing a brand-new stationary zone circle every ~360-500ms (their `cooldownMs`). Neither weapon's `updateZones()` cap existed at all (`zones.length` only appeared in the splice-cleanup loop, never checked against a limit) — confirmed via grep on both source and the live bundle (`MAX_ZONE`, `zoneCap`, "24" near any zone code: zero matches). With a 100-second lifetime and a fire cadence of roughly twice a second, a single continuous 15-kill play session accumulates on the order of 150-250 simultaneously-alive, never-expiring translucent circles at the player's trailing path — exactly the "massive concentric rings covering the entire screen" in the screenshot. Both weapons are designed as a following "aura" (per their own `description` fields), but the actual implementation only supports stamping down stationary zones on a cooldown, not a single position-tracking zone — `durationMs: 99999` looks like a leftover/placeholder value from whoever authored these two entries, never intended to ship.
38. **Fix, two parts**: (a) data — `durationMs` for skunk-cloud's 5 levels + evo-a, and purr-aura's 5 levels + its evo-a, changed from `99999` to `6000` (6s; within the 5-8s window requested). skunk-cloud's evo-b ("Musk Bomb") already had a sane `4000` and was left alone. (b) code — added a hard defensive cap in `WeaponSystem.spawnZone()`: `MAX_ZONES_PER_WEAPON = 20`; before pushing a new zone, force-expire (destroy graphic + shift) the oldest zone(s) until under the cap. This is deliberately independent of the duration fix — it's a backstop so a future mistuned duration/cooldown pair on *any* zone weapon (not just these two) can't blanket the screen again, regardless of what data ships.
39. **Verified live, functionally, not just by inspection.** Used `window.__understory` (the same debug hook as the camera-wrap fix) to: (1) grep the live bundle post-deploy and confirm zero `"durationMs":99999` occurrences and the exact new `durationMs:6e3` value inline for skunk-cloud; (2) grant the player `skunk-cloud` at level 5 in a live run and call the actual private `WeaponSystem.spawnZone`/`updateZones` methods directly (bypassing only the unrelated enemy-proximity fire-gate, which has nothing to do with this bug) to simulate 60 fires at a 500ms cadence (30 simulated seconds — 6x longer than the reported 15-kill session). Sampled `runtime.zones.length` every 5 fires: it climbed to 10-11 by the 5-second mark and then **stayed flat at 11 for the remaining 25 simulated seconds** — proof the duration fix alone keeps the live zone count bounded and stable, well under the 20-zone hard cap, instead of growing without limit like before.
